## 1. Visão Geral

### O Que Resolve

A moderação manual de anúncios não escala e a fraude erode a confiança no marketplace. Este plano introduz um sistema automático que analisa novos anúncios à procura de sinais de fraude — preço muito abaixo do mercado, fotos reutilizadas de outros anúncios, texto copiado, e padrões de múltiplos anúncios do mesmo criador num curto espaço de tempo. Anúncios sinalizados são **priorizados na fila de moderação**, com uma explicação do motivo, para que o administrador foque a atenção onde o risco é maior. A deteção é invisível para o utilizador final (só aparece no painel admin).

### Benchmark Competitivo

| Plataforma | Detecção de Fraude |
|---|---|
| **Standvirtual** | Manual |
| **AutoScout24** | Automática |
| **Webmotors (BR)** | Automática (verificação de anúncios) |
| **Mercado Livre (BR)** | Automática (reputação + anti-fraude) |
| **AutoTrader UK** | Automática |
| **OLX (PT/BR)** | Básico |
| **RecarGarage (atual)** | Não |
| **RecarGarage (proposto)** | Automática (score + fila priorizada) |

### Histórias de Utilizador

1. **Como administrador**, quero que o sistema sinalize automaticamente anúncios suspeitos (preço irrealista, fotos duplicadas, padrões conhecidos), para priorizar a minha revisão e proteger os utilizadores.
2. **Como administrador**, quero ver *porquê* um anúncio foi sinalizado (motivos + score), para decidir rapidamente entre confirmar a fraude ou marcar como falso positivo.
3. **Como comprador**, quero que a plataforma remova anúncios fraudulentos antes de eu ser enganado, mesmo sem saber que este sistema existe.

## 2. Especificação das Funcionalidades

### 2.1 Deteção Automática (Background)
Quando um novo anúncio é criado (`status: 'pendente'`), um trigger analisa-o e calcula um **score de suspeita (0-100)** a partir de vários sinais:
- **Preço vs. mercado:** preço significativamente abaixo da mediana de veículos semelhantes (reutiliza a lógica do Plano 02).
- **Fotos duplicadas:** hash percetual das fotos comparado com fotos de anúncios existentes.
- **Texto copiado:** similaridade do texto da descrição com outros anúncios (sobreposição de palavras-chave / TF-IDF básico).
- **Alta frequência:** número de anúncios do mesmo criador nos últimos 7 dias acima de um limiar.

Se o score exceder um limiar (ex.: > 50), cria-se um documento em `fraudAlerts` e marca-se `fraudeSuspect: true` no documento do anúncio.

### 2.2 Painel de Alertas no Admin
Nova aba **"Fraude"** no painel de administração com uma tabela de alertas ordenada por score:
- Colunas: Score (código de cores), Anúncio (link), Criador, Motivos (expansível), Data, Ações.
- Ações: **"Confirmar Fraude"** (remove o anúncio + notifica) e **"Falso Positivo"** (dispensa o alerta).
- Resumo de estatísticas: total de alertas, confirmados, taxa de falsos positivos.
- Na tabela geral de anúncios (`ListingsTable`), ícone de aviso (triângulo âmbar) nas linhas com `fraudeSuspect: true`, com tooltip a mostrar score e motivos.

### Fluxo (Background)
1. Novo anúncio submetido (`status: 'pendente'`).
2. Trigger `detectFraud` (`onDocumentCreated` em `cars/{carId}`) executa:
   a. Preço vs. média de mercado.
   b. Hash percetual das fotos vs. fotos existentes.
   c. Similaridade de texto com anúncios recentes.
   d. Frequência de anúncios do mesmo criador (7 dias).
3. Calcula o score global (0-100).
4. Se score > limiar → cria `fraudAlerts` + define `fraudeSuspect: true`, `fraudeScore`, `fraudeMotivos` no anúncio.
5. Admin vê os anúncios sinalizados com prioridade e explicação do motivo.

### Requisitos de UI/UX
- Deteção **invisível** para o utilizador final (só no painel admin).
- Score codificado por cor: 0-30 verde, 30-60 amarelo, 60-100 vermelho.
- Motivos apresentados de forma legível (ex.: "Preço 55% abaixo da média de mercado"), não como IDs crus.
- Ações destrutivas ("Confirmar Fraude" → remove anúncio) com confirmação.
- Reutilizar primitivos de `src/components/ui/` e padrões do painel admin existente.

### Coleção Firestore `fraudAlerts` (nova)
```typescript
interface FraudAlert {
  id: string;
  anuncioId: string;
  anuncioType: 'car' | 'part';
  criadorEmail: string;
  score: number;              // 0-100, quanto maior mais suspeito
  motivos: FraudMotivo[];
  status: 'pendente' | 'confirmado' | 'falso_positivo';
  resolvidoPor?: string;
  dataCriacao: Timestamp;
  dataResolucao?: Timestamp;
}

type FraudMotivo =
  | { tipo: 'preco_baixo'; detalhe: string; precoAnuncio: number; precoMedio: number }
  | { tipo: 'foto_duplicada'; detalhe: string; anuncioOriginalId: string }
  | { tipo: 'texto_copiado'; detalhe: string; similaridade: number; anuncioOriginalId: string }
  | { tipo: 'alta_frequencia'; detalhe: string; totalAnuncios: number; periodo: string }
  | { tipo: 'outro'; detalhe: string };
```

### Atualização nas coleções `cars` / `parts`
```typescript
{
  fraudeSuspect?: boolean;   // Flag de suspeita
  fraudeScore?: number;      // Score 0-100
  fraudeMotivos?: string[];  // Resumo legível dos motivos
}
```

### Regras de Segurança Firestore
```
match /fraudAlerts/{alertId} {
  allow read:   if isAdmin();
  allow create: if false;   // apenas Cloud Functions criam alertas
  allow update: if isAdmin();
  allow delete: if isAdmin();
}
```
> A escrita de `fraudeSuspect`/`fraudeScore`/`fraudeMotivos` no documento do anúncio é feita pela Cloud Function (Admin SDK), que ignora as regras. Nenhuma regra cliente adicional é necessária nos campos do anúncio.

### Detalhes de algoritmo (v1)
- **Preço:** consultar carros semelhantes (marca/modelo/ano) e comparar com a mediana; sinalizar se `preco < 50% da mediana`.
- **Fotos duplicadas:** hash percetual (ex.: pHash/aHash) das imagens; comparar com hashes de anúncios recentes; distância de Hamming baixa → duplicado.
- **Texto copiado:** sobreposição de palavras-chave / similaridade simples com descrições recentes.
- **Frequência:** contar anúncios do mesmo criador nos últimos 7 dias; sinalizar se `> 5`.
- **Score:** soma ponderada dos sinais, normalizada a 0-100. Limiar de alerta configurável (default 50).

## 5. Decisões de Arquitetura

### Decisão 1: Trigger vs. análise periódica

| Opção | Prós | Contras |
|---|---|---|
| **Trigger `onDocumentCreated`** | Análise imediata na criação; sinaliza antes da aprovação | Custo por escrita; picos se houver muitos anúncios simultâneos |
| **Job periódico (agendado)** | Amortiza carga; simples de rate-limitar | Latência (anúncio pode ficar pendente sem análise por minutos) |

**Recomendação:** **Trigger `onDocumentCreated`** para v1 — a análise chega antes do administrador rever o anúncio, que é o ponto onde o valor é maior. Se o volume crescer muito, migrar sinais mais caros (ex.: hashing de fotos) para um job periódico assíncrono.

### Decisão 2: Hashing de fotos — pHash local vs. serviço externo

| Opção | Prós | Contras |
|---|---|---|
| **pHash na Cloud Function** | Sem dependências externas; barato; suficiente para duplicados óbvios | Menos robusto a manipulações (crop, watermark) |
| **Serviço de reverse-image externo** | Deteta reutilização fora da plataforma | Custo; latência; API key externa |

**Recomendação:** **pHash local** na v1 (deteta o caso mais comum — a mesma foto reutilizada dentro da plataforma). Reverse-image externo fica para uma fase futura se justificar.
