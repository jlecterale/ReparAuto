Plano 26 — Avaliações Estruturadas (Reviews 2.0)

Prioridade: MÉDIA IMPLEMENTADO

Esforço: ~5-7 dias · PR #97

O Que Resolve

O sistema actual de avaliações é básico: 1-5 estrelas + texto. Não impede duplicatas a nível de base de dados, não permite edição pelo autor, e não diferencia critérios por tipo de anúncio. Isto reduz a credibilidade das avaliações e não ajuda compradores a tomar decisões informadas.

3. Base de Dados (src/lib/db.ts)

  • addReview — usa setDoc() com ID determinístico; auto-gera criterios se omissos
  • updateReview — qualquer edição re-pende a avaliação
  • getReviewById — leitura directa por ID determinístico
  • updateSellerRating — calcula média dos criterios (fallback nota)

4. Firestore Rules

match /reviews/{reviewId} {
  allow read: if true;
  allow create: if isAuthenticated() && isNotBanned()
    && request.resource.data.autorUid == request.auth.uid
    && request.resource.data.vendedorEmail != request.auth.token.email
    && request.resource.data.nota >= 1
    && request.resource.data.nota <= 5
    && request.resource.data.status == 'pendente';
  allow update: if (isAuthenticated()
    && resource.data.autorUid == request.auth.uid
    && request.resource.data.status == 'pendente')
    || isAdmin();
  allow delete: if isAuthenticated() && (resource.data.autorUid == request.auth.uid || isAdmin());
}

Decisões de Arquitetura

ID Determinístico vs Campo uniqueKey

Contexto: Como garantir 1 review por (autorUid, anuncioId) ao nível da base de dados?

Opções:

  • Campo uniqueKey + query — adicionar campo e consultar antes de criar (race condition)
  • ID determinístico com setDoc — usar autorUid_anuncioId como doc ID; setDoc falha se já existe

Escolha: ID determinístico. Garantia Firestore-level sem race conditions.

Edição: re-pendente vs apenas pendente

Contexto: O utilizador pode editar avaliações já aprovadas?

Escolha: Sim — qualquer edição re-pende a avaliação para re-moderação.
